# HUDS Active Learning App Project Plan

## 1. Project Goal

Build a standalone HUDS active-learning application for surrogate model training.

The app should let a user define:

- Variable names.
- Variable ranges.
- Sampling point count or sampling resolution per variable.
- Candidate pool size.
- Train/validation split policy.
- Model output dimension and output names.
- Hidden layer width.
- Number of residual blocks.
- Training loop settings.
- HUDS sampling settings.

The app should then support this workflow:

1. Initialize a stratified Latin hypercube candidate pool.
2. Split the candidate pool into training-candidate and validation-candidate lists.
3. Initialize a surrogate model from user-defined architecture parameters.
4. Export validation samples for real simulation.
5. Convert real simulation outputs into the app's validation data format.
6. Export initial training samples for real simulation.
7. Convert real simulation outputs into the app's training data format.
8. Train the model.
9. Use HUDS to select the next batch of samples from the remaining training-candidate pool.
10. Repeat simulation import, model training, and HUDS sampling until stopping criteria are met.

The first version should be file-based and CLI-driven. A web UI can be added later after the core workflow is stable.

## 2. Non-Goals For Version 1

- Do not embed any domain-specific physics simulator.
- Do not assume the original `codev4-7dim-optimization` project path.
- Do not hard-code variable names such as `lx`, `gap`, or `vx`.
- Do not hard-code output names such as `fx` and `fy`.
- Do not require CUDA, although CUDA should be supported.
- Do not require FAISS; use FAISS if installed and fall back to scikit-learn.
- Do not hide intermediate files. The user must be able to inspect CSV files at every step.

## 4. Configuration Design

The primary user input should be a JSON config file.

Example:

```json
{
  "project_name": "demo_huds_project",
  "random_seed": 42,
  "variables": [
    {"name": "lx", "min": 0.01, "max": 0.08, "sample_points": 12},
    {"name": "ly", "min": 0.01, "max": 0.10, "sample_points": 12}
  ],
  "candidate_pool": {
    "total_samples": 12000,
    "train_ratio": 0.8,
    "validation_ratio": 0.2
  },
  "model": {
    "output_names": ["fx", "fy"],
    "hidden_dim": 128,
    "residual_blocks": 4,
    "dropout": 0.1
  },
  "validation": {
    "default_size": 1000
  },
  "training": {
    "initial_train_size": 256,
    "sample_per_step": 64,
    "max_steps": 10,
    "epochs_per_step": 300,
    "batch_size": 256,
    "learning_rate": 0.001,
    "patience": 30,
    "device": "cuda"
  },
  "huds": {
    "pre_n": 0,
    "repeat_times": 30,
    "topk_ratio": 0.6,
    "batch_size": 256,
    "use_faiss": true
  }
}
```

Validation rules:

- `variables` must contain at least one variable.
- Variable names must be unique.
- `min < max` for every variable.
- `sample_points >= 1` for every variable.
- `candidate_pool.total_samples > 0`.
- `train_ratio + validation_ratio` should equal `1.0` within tolerance.
- `model.output_names` must contain at least one output.
- `hidden_dim > 0`.
- `residual_blocks >= 0`.
- `sample_per_step > 0`.
- `initial_train_size > 0`.
- `initial_train_size + sample_per_step * max_steps` should not exceed available training candidates unless the app handles early exhaustion.

## 5. Data Format Contract

All major files should be CSV files with UTF-8 encoding.

### 5.1 Candidate Pool

Generated by:

```bash
huds-app init --config config.json --out runs/demo
```

Output:

```text
runs/demo/candidate_pool.csv
```

Required columns:

```text
sample_id, split, status, <variable columns...>
```

Column meanings:

- `sample_id`: stable integer ID.
- `split`: one of `train_pool` or `validation_pool`.
- `status`: one of `unlabeled`, `selected`, `labeled`, `used`.
- variable columns: exactly the variable names from config.

Example:

```csv
sample_id,split,status,lx,ly,gap
0,train_pool,unlabeled,0.0123,0.0831,0.0092
1,validation_pool,unlabeled,0.0642,0.0194,0.0483
```

### 5.2 Training Candidate List

Output:

```text
runs/demo/train_pool.csv
```

Required columns:

```text
sample_id,<variable columns...>
```

This file contains all samples available for active-learning training selection.

### 5.3 Validation Candidate List

Output:

```text
runs/demo/validation_pool.csv
```

Required columns:

```text
sample_id,<variable columns...>
```

This file contains validation candidates before real simulation.

### 5.4 Validation Request File

Generated by:

```bash
huds-app export-validation --run runs/demo --size 1000
```

Output:

```text
runs/demo/requests/validation_request.csv
```

Required columns:

```text
sample_id,<variable columns...>
```

The user sends this file to the real simulator.

### 5.5 Simulator Output Format

The real simulator should return one CSV file with:

```text
sample_id,<variable columns...>,<output columns...>
```

For a model with `output_names = ["fx", "fy"]`, the simulator output should be:

```csv
sample_id,lx,ly,gap,fx,fy
10,0.0123,0.0831,0.0092,0.123,5.678
```

The app should support a conversion command:

```bash
huds-app import-labels \
  --run runs/demo \
  --kind validation \
  --input simulator_validation_output.csv
```

The command should verify:

- Every required variable column exists.
- Every required output column exists.
- `sample_id` exists and is unique.
- Values are finite numbers.
- `sample_id` belongs to the correct request file.

Converted output:

```text
runs/demo/datasets/validation_labeled.csv
```

Required columns:

```text
sample_id,<variable columns...>,<output columns...>
```

### 5.6 Training Request File

Generated by either initialization or HUDS sampling:

```bash
huds-app export-initial-train --run runs/demo
huds-app sample --run runs/demo --step 1
```

Output examples:

```text
runs/demo/requests/train_step_000_request.csv
runs/demo/requests/train_step_001_request.csv
```

Required columns:

```text
sample_id,<variable columns...>
```

After simulation, import labels:

```bash
huds-app import-labels \
  --run runs/demo \
  --kind train \
  --step 0 \
  --input simulator_train_step_000_output.csv
```

Converted output:

```text
runs/demo/datasets/train_labeled.csv
```

`train_labeled.csv` should be append-only by default. Duplicate `sample_id` rows should be rejected unless `--overwrite` is passed.

## 6. Stratified Latin Hypercube Sampling

The app should implement stratified Latin hypercube sampling without relying on SciPy.

For each variable:

1. Divide `[min, max]` into `N` strata, where `N = candidate_pool.total_samples`.
2. Draw one random value inside each stratum.
3. Randomly permute values for that variable.
4. Repeat for every variable.
5. Combine columns into an `(N, D)` candidate matrix.

How to use `sample_points`:

- Version 1 should support two modes.
- Default mode: `total_samples` controls LHS sample count; `sample_points` is metadata and can be used for optional snapping.
- Optional snapped mode: if `--snap-to-levels` is provided, each variable is first represented by `sample_points` evenly spaced levels, and LHS values are snapped to the nearest level.

Recommended CLI:

```bash
huds-app init --config config.json --out runs/demo
huds-app init --config config.json --out runs/demo --snap-to-levels
```

Split policy:

- Shuffle sample IDs with `random_seed`.
- Assign first `train_ratio` fraction to `train_pool`.
- Assign remaining fraction to `validation_pool`.
- Save all split information.

## 7. Model Design

Use a residual MLP because the input is tabular numeric data.

Model class:

```python
class ResidualMLP(nn.Module):
    def __init__(
        self,
        input_dim: int,
        output_dim: int,
        hidden_dim: int,
        residual_blocks: int,
        dropout: float,
    ):
        ...

    def forward(self, x, return_features: bool = False):
        ...
```

Requirements:

- Input dimension is `len(config.variables)`.
- Output dimension is `len(config.model.output_names)`.
- Hidden width is user-configurable.
- Residual block count is user-configurable.
- Dropout must be present so MC Dropout uncertainty works for HUDS.
- `return_features=True` should return `(prediction, features)`.
- Features should be the final hidden representation before the output head.

Normalization:

- Input normalization should use min/max from `candidate_pool.csv`.
- Target normalization should use mean/std from `train_labeled.csv`, or from validation labels if explicitly configured.
- Save normalization statistics to:

```text
runs/demo/artifacts/normalization.json
```

## 8. Validation Data Preparation

The app should let the user generate a validation set with a default size or manual size.

Command:

```bash
huds-app export-validation --run runs/demo
huds-app export-validation --run runs/demo --size 2000
```

Behavior:

- Select from `validation_pool.csv`.
- Default size is `config.validation.default_size`.
- If requested size exceeds available validation candidates, use all available candidates and warn.
- Mark selected validation samples in `state.json`.
- Write `requests/validation_request.csv`.

The validation labeled file should be used during model training and evaluation.

## 9. Training Data Preparation

Initial training request:

```bash
huds-app export-initial-train --run runs/demo
```

Behavior:

- Select `training.initial_train_size` samples from `train_pool.csv`.
- Selection should be random or LHS-order-preserving, controlled by a config option.
- Write `requests/train_step_000_request.csv`.
- Mark samples as selected but not labeled in `state.json`.

After simulation:

```bash
huds-app import-labels --run runs/demo --kind train --step 0 --input output.csv
```

The app appends imported rows to:

```text
runs/demo/datasets/train_labeled.csv
```

## 10. HUDS Algorithm

HUDS selection should follow:

1. Use current trained model.
2. Run MC Dropout prediction on candidate training pool.
3. Estimate uncertainty as mean predictive variance across output dimensions.
4. Keep top-k uncertain samples.
5. Standardize variable columns before clustering.
6. Cluster top-k samples with KMeans.
7. Select the highest-uncertainty sample from each cluster.
8. If there are fewer selected samples than requested, fill missing slots with k-center greedy on standardized feature space.

HUDS settings:

- `sample_per_step`: number of samples selected each active-learning step.
- `pre_n`: optional pre-sampling count. `0` means use the full unlabeled training pool.
- `repeat_times`: MC Dropout repeats.
- `topk_ratio`: fraction of candidates retained before clustering.
- `batch_size`: inference batch size.
- `use_faiss`: use FAISS KMeans if installed.

Command:

```bash
huds-app sample --run runs/demo --step 1
```

Input:

- `datasets/train_labeled.csv`
- `datasets/validation_labeled.csv`
- Latest model checkpoint.
- Remaining unlabeled rows in `train_pool.csv`.

Output:

```text
runs/demo/requests/train_step_001_request.csv
runs/demo/artifacts/huds_step_001.json
```

The JSON should include:

- selected sample IDs.
- uncertainty values.
- top-k size.
- number of clusters.
- cluster statistics.
- model checkpoint used.

## 11. Training Loop

Command:

```bash
huds-app train --run runs/demo
```

Training behavior:

1. Load config and state.
2. Load `train_labeled.csv`.
3. Load `validation_labeled.csv`.
4. Fit or load normalization stats.
5. Initialize model if no checkpoint exists.
6. Train for `epochs_per_step`.
7. Evaluate on validation set.
8. Save best checkpoint.
9. Save metrics.

Outputs:

```text
runs/demo/checkpoints/model_latest.pt
runs/demo/checkpoints/model_best.pt
runs/demo/metrics/training_metrics.csv
runs/demo/artifacts/normalization.json
```

Metrics CSV columns:

```text
step,epoch,train_loss,val_loss,val_r2_avg,val_r2_<output>,val_rmse_<output>,elapsed_s
```

Stopping criteria for active-learning loop:

- `step >= training.max_steps`.
- Training pool exhausted.
- Optional `val_r2_avg >= target_r2`.
- Optional no validation improvement for `patience` training evaluations.

## 16. Error Handling

The app should fail early with clear messages for:

- Missing config.
- Invalid config.
- Missing required CSV columns.
- Duplicate sample IDs.
- Non-finite numeric values.
- Importing labels for samples that were not requested.
- Training before validation labels exist.
- Training before any training labels exist.
- Sampling before a model checkpoint exists.
- Sampling when no unlabeled candidates remain.

## 17. Testing Plan

Minimum tests:

- Config loads and validates.
- Invalid variable ranges fail.
- Duplicate variable names fail.
- LHS candidate pool has expected shape and columns.
- Train/validation split is deterministic for a fixed seed.
- Simulator output schema validation catches missing outputs.
- Label import appends rows and rejects duplicates.
- Residual MLP returns correct prediction shape.
- `return_features=True` returns expected feature dimension.
- HUDS returns exactly `n_select` samples when enough candidates exist.
- HUDS returns fewer samples only when candidate pool is exhausted.
- CLI smoke test can initialize a tiny run and export request files.

## 21. Review Checklist For Future Audit

When reviewing the implementation, check:

- No domain-specific variable names are hard-coded.
- All file paths are relative to `--run` unless explicitly provided.
- The app preserves `sample_id` across every file.
- Simulator imports cannot silently reorder labels against samples.
- Training uses the same input normalization for train, validation, prediction, and HUDS.
- Target normalization is inverted before reporting human-readable metrics.
- HUDS uses model dropout mode only during MC uncertainty and restores model mode afterward.
- KMeans standardizes features before clustering.
- Random seeds are used consistently.
- Checkpoints include model config and normalization stats.
- CLI errors explain how to fix the problem.
